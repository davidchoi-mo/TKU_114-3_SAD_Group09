### D. 權力 / 關注矩陣

| 類型           | 本組利害關係人         | 後續需求蒐集策略                                                                             |
| -------------- | ---------------------- | -------------------------------------------------------------------------------------------- |
| 高權力、高關注 | 管理員                 | 優先確認系統功能、權限管理及異常處理流程，定期與管理者討論需求，確保系統符合管理需求。       |
| 高權力、低關注 | 校園管理處             | 確認校園配送規範、交接地點及取餐點設置是否符合校園管理要求，定期回報專案進度即可。           |
| 低權力、高關注 | 學生、店家、校內配送員 | 透過訪談、問卷或情境模擬蒐集需求，了解訂餐、接單、配送及取餐流程中的痛點，作為系統設計依據。 |
| 低權力、低關注 | 校外外賣員             | 確認交接流程及交接地點是否清楚即可，不需投入大量需求訪談，以基本通知及流程說明為主。         |

### E. 可行性分析

| 可行性類型                            | 分析內容                                                                                                                                                                                         | 判斷結果       | 需要調整的範圍                                         |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------- | ------------------------------------------------------ |
| 技術可行性（technical feasibility）   | 本專案可使用網頁技術或簡易資料庫模擬訂餐、接單、配送及取餐流程，不需串接真實金流或校務系統，因此以目前的技術能力可完成原型。不過，若加入即時定位、推播通知或第三方平台串接，技術難度將大幅提高。 | 可行，但有風險 | 不實作即時 GPS 定位、推播通知及第三方金流串接。        |
| 操作可行性（operational feasibility） | 系統符合學生、店家及配送員的實際使用情境，可整合訂餐、配送及取餐流程，降低人工聯繫與資訊分散的問題。不過，若未事先規劃校內固定交接點及取餐點，可能造成配送流程混亂，因此需先建立明確的配送規則。 | 可行           | 配送流程以固定交接點及指定取餐點為主，不規劃自由配送。 |
| 時程可行性（schedule feasibility）    | 本專案聚焦於學生訂餐、店家接單、校內配送及定點取餐三個核心流程，可在 5 週內完成分析、設計及原型展示。但若加入會員點數、AI 推薦、校外配送或完整後台管理等功能，將增加開發時間，影響專案進度。     | 可行，但有風險 | 僅完成核心流程原型，不實作進階功能與完整後台。         |
| 經濟可行性（economic feasibility）    | 本專案以課程原型為目標，可使用 GitHub 及其他免費開發工具，不需額外購買伺服器或商業軟體，因此成本較低。若未來正式導入校園，才需考量伺服器建置、系統維護及人力成本。                              | 可行           | 本期僅完成原型，不考慮正式部署及維運成本。             |

### G. 今天新增的範圍外項目

1. 不提供訂單評價、評論及店家評分功能。
2. 不提供優惠券、會員點數或促銷活動管理功能。
3. 不提供即時聊天功能，使用者以系統通知進行訂單與取餐資訊確認。
